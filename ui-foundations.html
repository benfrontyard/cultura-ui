UI Foundations

A simple guide to how Culture Hive looks and behaves: colors, type, icons, spacing, and common UI pieces. If you are not writing code, use it to see what “official” patterns look like. If you are building screens, everything here matches the live product because it uses the same stylesheets as the rest of the app.

Built on Bootstrap 5.3 with a Culture Hive layer in theme.css on GitHub.

How updates work

When designers or developers change the product look, they change shared CSS files, not just this page. The real app and this reference both load the same bundle, so what you see here is what users get after a refresh.

  • For everyone: think of theme.css as the master style sheet for Culture Hive (colors, type, cards, tables, and more).
  • For developers: brand tokens live as --ch-* variables in :root. Extra names like --color-primary-500 or --surface-card are thin aliases that point at those same --ch-* values so docs and code can speak the same language.
  • Only on this page: ui-foundations.css handles the sidebar table of contents and layout of examples, not the product UI itself.
  • Sign-in gate: the log-in and sign-up gate on index.html uses the same theme.css primitives as the rest of the app (.ch-auth-* wrappers plus .form-control, .btn-primary, and shared type classes). Treat the Sign in specimen below as the documented markup pattern; do not fork a second visual system for authentication screens.

Icons

Icons are simple line drawings (not photos). They inherit the surrounding text color so they stay on-brand in light UI and in buttons. In the product we use one shared set so navigation, alerts, and cards all feel consistent.

  • Where they come from: the artwork is based on the open Streamline vectors family on GitHub (third-party library).
  • Culture Hive copy in this repo: the SVG “sprite” (one file with every icon) is assets/icons/streamline-sprite.svg (each icon has an id like ch-house). A vendored upstream checkout also lives under vendor/streamline-vectors/. Sizing classes are in icons.css (.ch-icon, .ch-icon--sm, .ch-icon--xl, .ch-icon--2xl).
  • For developers: when you open HTML from disk, ch-icon-sprite-inject.js inlines the sprite so icons still render; regenerate it with node scripts/sync-ch-icon-sprite-inject.mjs after editing the sprite.

Typography

Fonts you will see: Epilogue for big marketing-style titles, General Sans for most UI text and smaller headings (loaded via Fontshare). Technical sizing follows a clear step scale inspired by Shopify Polaris type tokens (good reference for designers); in code those steps are --ch-font-size-* and --ch-font-line-height-*. Fluid sizes --ch-text-2xs through --ch-text-md plus a responsive root font size keep things readable on phones and desktops.

Heading weight: large titles use semibold (600), not extra-heavy bold. Inline labels and emphasis in body copy use the same semibold (600) via the global strong rule in theme.css.

Keyboard focus

When someone tabs through the interface, focused buttons and links get a clear blue ring so keyboard users know where they are. That behavior comes from shared focus styles in the theme, aligned with :focus-visible on MDN.

Buttons

Primary actions use solid blue; secondary actions use gray; destructive work uses red. Ghost-style buttons are light with a thin border so they feel quieter on dense screens. Under the hood these are standard Bootstrap buttons with Culture Hive colors wired in theme.css.

Forms

Labels, inputs, selects, and validation messages follow Bootstrap form patterns. Error states use Bootstrap’s invalid styles plus Culture Hive colors. Dense public surfaces (the Assistant gate) add .ch-auth-* spacing and label styles on top of the same controls; see Sign in.

Sign in and account creation

Culture Hive opens with the same split layout you see on index.html: media on the left, form on the right at desktop widths. All styling lives in theme.css under the .ch-auth-* prefix; this page does not ship a separate auth stylesheet.

  • Title: .ch-auth-title with a type ramp class such as .ch-type-heading-xl.
  • Supporting line: .ch-auth-lead (uses body-large tokens).
  • Fields: .ch-auth-field, .ch-auth-label or .ch-auth-label-row, .form-control.ch-auth-input; password fields pair .ch-auth-input-wrap, .ch-auth-input--toggle, and .ch-auth-password-toggle with Streamline #ch-eye / #ch-eye-off icons.
  • Actions: .btn.btn-primary.ch-auth-submit for the primary action; secondary switches use .ch-auth-text-link (button element, underlined).
  • Footer chrome: .ch-auth-hint, .ch-auth-page-footer, .ch-auth-footer-link for legal and meta lines.

Modals and toasts

Modals pause the page for a focused task; toasts confirm actions. Shell modals (#chHelpModal, #chUpgradeModal, #chAudienceModal) use identical copy on every workspace page. Demo actions use data-ch-demo-toast with a specific data-ch-toast-message — never generic fallbacks.

Navigation

The left rail lists main areas of the product; the current page is highlighted automatically. Each screen sets a simple marker on the <body> tag; JavaScript in app.js matches that marker to the correct nav link. On small screens the same links appear inside a slide-out menu. Back and forward in the top bar follow normal browser history. Developers: look for data-ch-current on the page, data-ch-nav on links, optional <details class="ch-nav-group"> for grouped items, and initTopbarHistoryNav for the history buttons.
